jASL Source Browser

Building jASL

The generation of the files and documentation for the jASL program requires GNU Make, as well as other standard UNIX utilities. Alternatives for build automation are left as an exercise.

The following conditions are assumed about the system used to process the files in the source directory and its sub-directories:

  • A Java SE SDK is installed.
  • The original directory structure for this system is in place. The top level directory is jasl, which is referred to as $JASL_BASE in this document. An environment variable, which is necessary for make to work properly, can be set as follows: export JASL_BASE=/home/craig/jasl

  • Optional. GCC (using another C++ compiler is possible, but doing so will (obviously) require changes to the build system). If it is used to build the libraries that use the JNI to interact with the Java bytecode, it must be C++17 compliant. Setting the JAVA_HOME environment variable (e.g. export JAVA_HOME=/opt/java) is also necessary in order to find the JNI headers and library.
  • Optional. GCJ (available with GCC version 6.5 and earlier). It is used to build (from Java .class files) the native executable versions of this program, static and shared libraries, and header files. Note that it requires compilation of the Java source files with JDK 8 or older.
  • Optional. SWIG, which is used, along with GCC (6.5 and earlier), to build wrapper modules and libraries that allow access to the (GCJ generated) shared libraries by scripting languages such as perl and python.
  • Optional. GNU Source-highlight. This program is used to create HTML versions of the source files.
  • Optional. Perl and GNU sed, which are used to build a linked hierarchy of the HTML versions of the source files.
  • Optional. Doxygen.
  • Optional. Exuberant Ctags.
  • Optional. GNU GLOBAL. Using this program requires the following environment variables to be set:
      export GTAGSROOT=$JASL_BASE/source
      export GTAGSDBPATH=$GTAGSROOT/gtags

The function of each target defined in the Makefile is described below. The executables and scripts are written or copied to (and deleted from) the output directory ($JASL_BASE/bin) and its sub-directories. All of the libraries referenced are found in $JASL_BASE/lib/jasl and its sub-directories.

  • none or all - compiles all of the .java source files using javac, placing the java bytecode (.class) files in the output directory.

  • build_jar - create an executable jar archive of the jASL program.

  • build_gcj - executes the actions defined for the gcj target. Upon completion, the clean_obj, clean_class, and clean_java targets are executed, removing all of the intermediate files created by the build.

  • build_gcj_dynamic - executes the actions defined for the gcj_dynamic target. Upon completion, the clean_class and clean_java targets are executed, removing all of the intermediate files created by the build.
  • build_gcj_static - performs a similar function to the build_gcj_dynamic target, with the gcj_static target replacing gcj_dynamic.

  • build_cpp_dynamic - compiles the .cpp source file(s) using gcj and links them with shared versions of the jasl libraries and the cni-wrapper library. These native executable(s) have a "_cpp_dynamic" suffix and are placed in the output directory.
  • build_cpp_static - performs a similar function to the build_cpp_dynamic target, except the linking is with the static libraries and each output filename ends with "_cpp_static".

  • build_cxx_dynamic - compiles the .cxx source file(s) using gcc (or another C++ compiler) and links them with shared versions of the jni-jasl and jni-wrapper libraries. These native executable(s) have a "_cxx_dynamic" suffix and are placed in the output directory.
  • build_cxx_static - performs a similar function to the build_cxx_dynamic target, except the linking is with the static libraries and each output filename ends with "_cxx_static".

  • obj - compiles all of the .class files generated within the jasl source hierarchy using gcj, placing the object (.o) files in corresponding .obj sub-directories.
  • gcj - executes the actions defined for the obj target and then builds native executable(s), linked with the individual object (.o) files, using gcj. The file(s) are placed in the output directory.
  • gcj_dynamic - executes the actions defined for the all target. When this is complete, native executable(s), linked with the shared jasl libraries, are generated using gcj. The file(s), which have a "_dynamic" suffix, are placed in the output directory.
  • gcj_static - performs a similar function to the gcj_dynamic target, except the linking is with the static jasl libraries and each output filename ends with "_static".

    The following targets install scripts that require modules and libraries generated with SWIG (see here for details), which link with the shared versions of the jasl libraries and the cni-wrapper library. The scripts are copied to the indicated sub-directories of the output directory.

  • perl - perl-jasl
  • python - py-jasl

    The clean targets remove the specified files from the output directory and/or its sub-directories unless otherwise noted.

  • clean_all - executes the actions defined for the clean_java, clean_obj, and clean targets.

  • clean_java - the jasl (output) sub-directory (i.e. all of the package bytecode (.class) files).
  • clean_obj - all of the .obj directories within the jasl source hierarchy.

  • clean - executes the actions defined for each of the clean_xxx targets defined below.

  • clean_class - the java bytecode (.class) file(s).
  • clean_jar - the jar archive.
  • clean_gcj - the native (from .class bytecode and linked with individual object (.o) files) executable(s).
  • clean_gcj_dynamic - the native (from .class bytecode and linked with the shared jasl libraries) executable(s).
  • clean_gcj_static - the native (from .class bytecode and linked with the static jasl libraries) executable(s).
  • clean_cpp_dynamic - the native (from .cpp source and linked with the shared versions of the jasl libraries and the cni-wrapper library) executable(s).
  • clean_cpp_static - the native (from .cpp source and linked with the static versions of the jasl libraries and the cni-wrapper library) executable(s).
  • clean_cxx_dynamic - the native (from .cxx source and linked with the shared versions of the jni-jasl and jni-wrapper libraries) executable(s).
  • clean_cxx_static - the native (from .cxx source and linked with the static versions of the jni-jasl and jni-wrapper libraries) executable(s).
  • clean_perl - the perl script(s) (see above for sub-directory name).
  • clean_python - the python script(s) (see above for sub-directory name).

    The test targets execute a program in the specified output or file format and compare the results, using diff, to a saved file.

  • test - the java bytecode (.class) file(s).
  • test_jar - the jar archive.
  • test_gcj - the native executable(s), linked with individual object (.o) files.
  • test_gcj_dynamic - the native (from .class bytecode) executable(s), linked with the shared jasl libraries.
  • test_gcj_static - the native (from .class bytecode) executable(s), linked with the static jasl libraries.
  • test_cpp_dynamic - the native (from .cpp source) executable(s), linked with the shared versions of the jasl libraries and the cni-wrapper library.
  • test_cpp_static - the native (from .cpp source) executable(s), linked with the static versions of the jasl libraries and the cni-wrapper library.
  • test_cxx_dynamic - the native (from .cxx source) executable(s), linked with the shared versions of the jni-jasl and jni-wrapper libraries.
  • test_cxx_static - the native (from .cxx source) executable(s), linked with the static versions of the jni-jasl and jni-wrapper libraries.
  • test_perl - the perl script(s).
  • test_python - the python script(s).

    The following targets are used to generate the documentation from the source files, placing the results in the indicated sub-directory of $JASL_BASE/share/jasl/doc.

  • docs - generates an HTML version of each source file and the Makefile using source-highlight, placing the output files in source. This file, source.html, is also generated, and includes links to all of the HTML files in their respective sub-directories.
  • javadocs - generate class documentation for this program using javadoc, placing the output files in javadocs.
  • doxygen - generates class documentation for this program using doxygen, placing the output files in doxygen/html.
  • htags - generates a hypertext version of the source code using global, allowing browsing by object and symbol names. The global target (see below) is used to rebuild the tag data necessary for input, and the output files are written to global/HTML.

  • all_docs - executes the actions defined for the docs, javadocs, doxygen, and htags targets.

    Miscellaneous targets.

  • scenarios - copies the JSON files in the sub-directory with the same name to $JASL_BASE/share/jasl/data/scenarios. Note that this target must be executed in order for the test targets to function correctly.

  • ctags - creates a file (tags) in the source directory that indexes many of the language objects. This is useful for navigating the source code from text editors, such as Vim, that support it.

  • global - creates a sub-directory (gtags) of the source directory containing tag files, which can be used by a number of programs to provide source navigation functionality. They are also used by htags (see above).
